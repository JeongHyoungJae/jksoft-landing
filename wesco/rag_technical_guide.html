RAG 벡터 검색 기술 구현 가이드

AI 기반 사이트 검색 시스템의 기술적 구현 방안과 벡터 데이터 관리 전략을 정리합니다.

1. RAG(Retrieval-Augmented Generation) 개요

RAG는 사이트 내부 데이터만을 기반으로 AI가 답변을 생성하는 기술입니다. 외부 지식에 의존하지 않으므로 정확도가 높고, 토큰 비용이 최소화됩니다.

작동 흐름

사용자 질문 입력
    ↓
① 질문을 벡터(Embedding)로 변환 — Gemini API
    ↓
② 벡터 DB에서 유사도 검색 — pgvector (PostgreSQL)
    ↓
③ 관련 콘텐츠 청크 추출 (상위 3~5개)
    ↓
④ 질문 + 관련 콘텐츠를 LLM에 전달 — Gemini API
    ↓
⑤ 사이트 정보 기반 답변 생성 → 사용자에게 응답

💡 핵심: LLM은 사이트 콘텐츠만을 context로 받기 때문에, 할루시네이션(허위 답변) 위험이 낮고 "우리 사이트에 없는 정보"는 답변하지 않도록 제어 가능합니다.

2. 벡터 데이터 생성 및 업데이트 주기

시점 작업 내용 비용 빈도
초기 구축 기존 사이트 전체 콘텐츠(기술문서, FAQ, 제품정보 등)를 벡터로 일괄 변환 1회성 (초기 구축비에 포함) 1회
콘텐츠 추가/수정 시 해당 글만 벡터로 변환하여 DB에 추가 (증분 업데이트) 건당 0.01원 이하 글 저장 시 자동
콘텐츠 삭제 시 해당 벡터 데이터 DB에서 제거 무료 (DB 삭제) 글 삭제 시 자동
전체 재인덱싱 임베딩 모델 변경, 청크 전략 수정 등 구조적 변경 시 소량 (전체 문서 수에 비례) 연 1~2회 이하

⚠️ "최초 1회만 생성"은 아닙니다. 다만, CMS에서 글 저장 시 자동으로 해당 글만 벡터 변환되므로 운영자가 별도로 신경 쓸 일은 없습니다.

방식 설명 추가 비용 적합성

추천 아키텍처: PostgreSQL + pgvector

별도 벡터 DB 서버 없이 기존 NHN Cloud PostgreSQL 인스턴스 하나로 해결합니다.

PostgreSQL + pgvector의 개념 및 장점

pgvector는 전 세계적으로 가장 널리 사용되는 오픈소스 관계형 데이터베이스인 PostgreSQL에 벡터 저장 및 검색 기능을 추가하는 공식 확장 플러그인(Extension)입니다.

  • 운영 복잡성 제로(Zero Complexity): 기존에 텍스트, 사용자 정보, 결제 내역 등을 저장하던 RDBMS(관계형 데이터베이스) 내에 벡터 데이터를 함께 저장할 수 있습니다. 즉, 별도의 전용 벡터 DB(Pinecone, Milvus 등)를 구축/배포/백업/모니터링할 필요가 없어 인프라 관리 비용이 획기적으로 절감됩니다.
  • 하이브리드 검색(Hybrid Search) 최적화: 실제 비즈니스 환경에서는 단순한 "문장 유사도" 검색만으로 부족한 경우가 많습니다. 예를 들어 "최근 3개월 이내에 작성된 기술 문서 중에서 '정전 보상'과 관련도가 높은 순서대로 찾아줘" 와 같은 복합 질의가 필요합니다. pgvector를 사용하면 SQL 쿼리 하나로 WHERE date > '2023-01-01' (관계형 필터링)과 ORDER BY embedding <-> '[...]' (벡터 유사도 검색)을 동시에 완벽하게 처리할 수 있습니다.
  • 데이터 정합성 보장(ACID Compliance): 게시글 원본 데이터와 벡터 데이터가 동일한 데이터베이스 내에 존재하므로 트랜잭션 관리가 용이합니다. 게시글이 삭제되면 동시에 해당 벡터 데이터도 확실하게 삭제(Cascade 제약조건 등 활용)되어 "고아(Orphan) 벡터"가 발생하는 문제를 원천 차단합니다.
  • NHN Cloud 완벽 호환: NHN Cloud를 비롯한 대부분의 클라우드 서비스 제공자(CSP)가 제공하는 Managed PostgreSQL 환경에서 pgvector 확장을 기본적으로 지원하거나 손쉽게 활성화할 수 있어 파편화 없이 메인 서버 아키텍처에 스며들 수 있습니다.
NHN Cloud 서버 인프라
├── NestJS API 서버
│   ├── /api/search — AI 검색 엔드포인트

    └── Gemini API
        ├── 임베딩 생성 (text-embedding-004)
        └── 답변 생성 (gemini-2.0-flash)

4. 토큰 비용 상세 분석

검색 1건당 비용 (Gemini 2.0 Flash 기준)

단계 사용 API 토큰 수 비용 (원)
질문 임베딩 text-embedding-004 ~50 토큰 약 0.001원
답변 생성 (입력) gemini-2.0-flash ~1,500 토큰 (질문 + context) 약 0.02원
답변 생성 (출력) gemini-2.0-flash ~500 토큰 약 0.03원
합계 (검색 1건) 약 0.05원

월간 비용 시뮬레이션

일 검색 횟수 월 검색 횟수 월 예상 비용
50건/일 1,500건 약 75원
100건/일 3,000건 약 150원
500건/일 15,000건 약 750원

5. CMS 자동화 연동 흐름

게시물 저장 시 자동 벡터 생성

관리자가 CMS에서 글 저장
    ↓
① 본문 텍스트를 의미 단위로 분할 (Chunking)
   - 단락 기준 300~500자 단위
   - 제목/소제목 컨텍스트 유지
    ↓
② 각 청크를 Gemini Embedding API로 벡터 변환
    ↓
③ 벡터 데이터를 PostgreSQL embeddings 테이블에 저장
    ↓
④ 동시에 AIO 메타데이터(스키마, 요약) 자동 생성
    ↓
✅ 완료 — 즉시 AI 검색에 반영

검색 품질 개선 전략

  • 하이브리드 검색: 벡터 유사도 검색 + 키워드 전문 검색(Full-Text Search)을 결합하여 정확도 향상
  • 메타데이터 필터링: 카테고리, 제품군, 기술 분야별 필터를 적용하여 검색 범위 제한
  • 답변 출처 표시: AI 답변에 원본 문서 링크를 함께 제공하여 신뢰도 확보
  • 피드백 루프: 사용자의 "도움이 됐어요/아니요" 반응을 수집하여 검색 품질 지속 개선
